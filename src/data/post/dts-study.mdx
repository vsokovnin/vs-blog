## Что такое изучение в DTS

Изучение — это второй этап уровня бизнес-модели в методологии **дизайн-мышления для стратегии (Design Thinking for Strategy, DTS)**. На предыдущем этапе (наблюдение) команда собрала сырые инсайты. Теперь задача — превратить их в структурированные знания.

### Место в процессе дизайн-мышления для стратегии

```
ФУНДАМЕНТАЛЬНЫЙ УРОВЕНЬ
Б:Бриф → С:Среда → Т:Тренды → Ф:Фокус
                                  │
                          КТ1: Утверждение фокуса
                                  │
                                  ▼
УРОВЕНЬ БИЗНЕС-МОДЕЛИ
Н:Наблюдение → И:ИЗУЧЕНИЕ → П:Проектирование → В:Валидация
                   ▲                              │
                   └──────── итерации ────────────┘
```

Процесс изучения (И) — конвергентный этап. Если наблюдение — это дивергенция (собрать как можно больше), то изучение — конвергенция (отфильтровать, структурировать, приоритизировать). Результаты изучения передаются на этап проектирования бизнес-модели.

### Изучение vs анализ данных

Картирование инсайтов в DTS — это не традиционный data-анализ. Ключевые отличия:

| Анализ данных | Изучение в DTS |
|---------------|----------------|
| Количественные метрики | Качественные инсайты |
| Статистические паттерны | Смысловые кластеры |
| Корреляции и тренды | Причинно-следственные связи |
| Ответ на «что происходит» | Ответ на «почему это важно для стратегии» |
| Результат: отчёт с цифрами | Результат: знания для проектирования |

Согласно статье «[Методологии проектирования стратегии бизнеса](https://cyberleninka.ru/article/n/metodologii-proektirovaniya-strategii-biznesa-ot-dizayna-produkta-k-proektirovaniyu-platform)», переход от инсайтов к знаниям требует не только структурирования данных, но и их интерпретации в контексте бизнес-модели.

### Почему картирование критично для стратегии

Без систематического картирования инсайты остаются разрозненными наблюдениями. По данным [McKinsey](https://www.mckinsey.com/capabilities/growth-marketing-and-sales/our-insights/the-new-consumer-decision-journey), компании, которые системно связывают клиентские инсайты с бизнес-процессами, повышают ROI маркетинга до 15%.

Проблема многих компаний: исследования проводятся, отчёты пишутся, но инсайты не превращаются в действия. Картирование решает эту проблему, связывая каждый инсайт с конкретным элементом бизнес-модели.

## Четыре шага процесса изучения

Процесс изучения в DTS состоит из четырёх последовательных шагов.

```
И:1 ─── Синтез инсайтов
        │
        └── Кластеризация и категоризация наблюдений

И:2 ─── Отображение на бизнес-модель
        │
        └── Связь инсайтов с элементами детализированной
            бизнес-модели

И:3 ─── Идентификация паттернов
        │
        └── Выявление закономерностей и противоречий

И:4 ─── Формулирование знаний
        │
        └── Документирование ключевых знаний для проектирования
```

### Шаг И:1 — Синтез инсайтов

На первом шаге команда собирает все инсайты из наблюдений и группирует их по схожести. Это основа для дальнейшего анализа.

**Процесс синтеза:**

1. **Собрать все инсайты.** Выписать каждый инсайт на отдельный стикер (физический или виртуальный). Один инсайт — один стикер.

2. **Кластеризовать по схожести.** Разложить стикеры на столе и сгруппировать похожие. Не планировать категории заранее — дать им проявиться.

3. **Назвать каждый кластер.** Придумать ёмкое название, отражающее суть группы. Название должно быть понятным без контекста.

4. **Определить отношения между кластерами.** Какие кластеры связаны? Какие противоречат друг другу? Визуализировать связи.

**Пример кластеризации для B2B SaaS:**

```
КЛАСТЕР 1: «Страх потери данных»
- Инсайт: Анна фотографирует тетрадь каждый вечер
- Инсайт: Сергей боится, что менеджер уйдёт с базой
- Инсайт: Михаил делает бэкапы в Excel раз в неделю

КЛАСТЕР 2: «Нет времени на обучение»
- Инсайт: «Пока разберусь — клиент уйдёт»
- Инсайт: Интерфейс должен быть «как WhatsApp»
- Инсайт: Видеоинструкции не смотрят

КЛАСТЕР 3: «WhatsApp как центр коммуникации»
- Инсайт: Все общаются с клиентами через WhatsApp
- Инсайт: История переписки — единственный «архив»
- Инсайт: Не хотят «ещё одно приложение»
```

**Принцип:** Знания выходят за пределы клиентов. Синтезируйте инсайты не только о клиентах, но и о компетенциях, технологиях, каналах, затратах.

### Шаг И:2 — Отображение на бизнес-модель

После синтеза каждый инсайт связывается с элементами детализированной бизнес-модели. Это ключевой шаг, который превращает абстрактные наблюдения в конкретные направления для проектирования.

**Элементы детализированной бизнес-модели (Business Model Canvas):**

| Код | Элемент | Вопрос |
|-----|---------|--------|
| КС | Клиентские сегменты | Для кого создаём ценность? |
| КЗ | Каналы (заказ) | Как клиенты узнают о нас и покупают? |
| КВ | Взаимоотношения с клиентами | Какие отношения ожидают клиенты? |
| КД | Каналы (доставка) | Как доставляем ценность? |
| ПЦП | Ценностное предложение | Какую ценность доставляем? |
| ППУ | Потоки дохода | За что клиенты платят? |
| КА | Ключевые активности | Что делаем для создания ценности? |
| КР | Ключевые ресурсы | Какие ресурсы нужны? |
| ФЗ | Структура затрат | Основные затраты? |
| ФД | Ключевые партнёры | Кто помогает создавать ценность? |

**Матрица отображения инсайтов:**

| Инсайт | КС | ПЦП | КВ | КЗ | КД | КА | КР |
|--------|----|----|----|----|----|----|-----|
| Страх потери данных | ✓ | ✓ | | | | ✓ | |
| Нет времени на обучение | ✓ | ✓ | ✓ | | | | |
| WhatsApp как центр | ✓ | | ✓ | ✓ | ✓ | | ✓ |

По данным [vc.ru](https://vc.ru/marketing/575262-customer-development-ili-kastdev-custdev-v-biznese-instrukciya-s-primerami), 70% стартапов не доходят до связывания инсайтов с бизнес-моделью, что приводит к разрыву между исследованиями и продуктом.

### Шаг И:3 — Идентификация паттернов

Третий шаг — поиск закономерностей, противоречий и пробелов в картированных инсайтах.

**Что искать:**

1. **Повторяющиеся темы.** Какие проблемы/потребности упоминаются чаще всего? Это сигнал о критичности.

2. **Противоречия между инсайтами.** Один сегмент хочет простоту, другой — функциональность. Это сигнал о необходимости сегментации.

3. **Неожиданные связи.** «Страх потери данных» связан с «нет времени на обучение» — клиенты боятся сложных систем, потому что не смогут восстановить данные при ошибке.

4. **Пробелы.** Какие элементы бизнес-модели не покрыты инсайтами? Возможно, нужно вернуться к наблюдению.

**Пример анализа паттернов:**

```
ПАТТЕРН 1: Противоречие
- Инсайт А: «Хотят интеграцию с WhatsApp»
- Инсайт Б: «Не хотят ещё одно приложение»
→ ВЫВОД: Нужно не «интеграция», а работа ВНУТРИ WhatsApp

ПАТТЕРН 2: Неожиданная связь
- Инсайт: «Фотографирует тетрадь на телефон»
- Инсайт: «Надиктовать голосом, а система разберётся»
→ ВЫВОД: Телефон — привычный интерфейс, голос — привычный ввод

ПРОБЕЛ: Нет инсайтов о ценовой чувствительности
→ ДЕЙСТВИЕ: Вернуться к наблюдению, добавить вопросы о бюджете
```

### Шаг И:4 — Формулирование знаний

Финальный шаг — документирование ключевых знаний в формате, пригодном для проектирования бизнес-модели.

**Формат документирования знания:**

```
ЗНАНИЕ #1: Клиенты боятся сложных систем из-за риска потери данных

ИНСАЙТЫ-ИСТОЧНИКИ:
- Анна фотографирует тетрадь каждый вечер
- Сергей: «Уходит менеджер — уходит база»
- Михаил делает бэкапы раз в неделю

СВЯЗЬ С БИЗНЕС-МОДЕЛЬЮ:
- КС (Клиентские сегменты): Малый бизнес без IT-специалиста
- ПЦП (Ценностное предложение): «Данные в безопасности без усилий»
- КА (Ключевые активности): Автоматическое резервное копирование

СЛЕДСТВИЯ ДЛЯ СТРАТЕГИИ:
- Простота важнее функциональности
- Безопасность данных — must have, не nice to have
- Onboarding должен включать демонстрацию восстановления данных
```

**Критерии качества знания:**

| Критерий | Описание | Проверка |
|----------|----------|----------|
| **Actionable** | Можно использовать для проектирования | Что конкретно изменить в продукте/процессе? |
| **Evidenced** | Основано на инсайтах, не на мнениях | Какие инсайты подтверждают? |
| **Connected** | Связано с бизнес-моделью | К каким элементам относится? |
| **Non-obvious** | Не очевидно без исследований | Знали бы это без наблюдений? |

По данным исследования [McKinsey](https://www.mckinsey.com/capabilities/growth-marketing-and-sales/our-insights/the-new-consumer-decision-journey), компании, формулирующие знания в формате «инсайт → следствие для бизнеса», в 2.4 раза чаще успешно внедряют результаты исследований.

## Практический пример: картирование для B2B SaaS

Продолжим пример из предыдущей статьи о наблюдении. CRM-система для малого бизнеса со стратегическим фокусом на клиентах.

**Исходные инсайты из наблюдения (20+ инсайтов):**

После наблюдений у Анны (салон красоты), Сергея (дистрибьютор), Михаила (автосервис) и фокус-групп собрано 24 инсайта.

**И:1 — Синтез инсайтов (результат кластеризации):**

```
КЛАСТЕР А: «Страх потери данных» (5 инсайтов)
КЛАСТЕР Б: «Нет времени на обучение» (4 инсайта)
КЛАСТЕР В: «WhatsApp как центр» (6 инсайтов)
КЛАСТЕР Г: «Контроль сотрудников» (3 инсайта)
КЛАСТЕР Д: «Напоминания клиентам» (4 инсайта)
КЛАСТЕР Е: «Не понимают термин CRM» (2 инсайта)
```

**И:2 — Отображение на бизнес-модель:**

| Кластер | КС | ПЦП | КВ | КЗ | КД | КА | КР |
|---------|----|----|----|----|----|----|-----|
| Страх потери данных | ✓ | ✓ | | | | ✓ | ✓ |
| Нет времени на обучение | ✓ | ✓ | ✓ | | | | |
| WhatsApp как центр | ✓ | | ✓ | ✓ | ✓ | | ✓ |
| Контроль сотрудников | ✓ | ✓ | | | | ✓ | |
| Напоминания клиентам | | ✓ | ✓ | | ✓ | ✓ | |
| Не понимают «CRM» | | | | ✓ | | | |

**Вывод из матрицы:** Максимальное покрытие — ПЦП и КС. Пробел — ФЗ и ФД (нет инсайтов о затратах и партнёрах).

**И:3 — Паттерны:**

```
ПАТТЕРН 1: Конфликт «Контроль vs Простота»
Владельцы хотят контролировать сотрудников, но сами не хотят
разбираться в сложных системах.
→ ВЫВОД: Разные интерфейсы для владельца и сотрудника

ПАТТЕРН 2: WhatsApp — не канал, а платформа
Клиенты не хотят интеграцию С WhatsApp, они хотят работать В WhatsApp.
→ ВЫВОД: Рассмотреть WhatsApp Business API как основу

ПАТТЕРН 3: «CRM» — токсичный термин
Малый бизнес ассоциирует CRM со сложностью и высокой ценой.
→ ВЫВОД: Позиционирование как «база клиентов», не «CRM»
```

**И:4 — Сформулированные знания (3 из 7):**

```
ЗНАНИЕ #1: Безопасность данных — критичнее функциональности
ИНСАЙТЫ: [5 инсайтов из кластера А]
БИЗНЕС-МОДЕЛЬ: ПЦП, КА, КР
СЛЕДСТВИЯ:
- Автобэкап каждый час (не день)
- Onboarding: показать восстановление за 1 клик
- Маркетинг: «Ваши данные никогда не потеряются»

ЗНАНИЕ #2: WhatsApp — единственный допустимый интерфейс
ИНСАЙТЫ: [6 инсайтов из кластера В]
БИЗНЕС-МОДЕЛЬ: КВ, КЗ, КД, КР
СЛЕДСТВИЯ:
- MVP через WhatsApp Business API
- Отказ от standalone-приложения
- Ввод данных: голосовые сообщения + OCR фото

ЗНАНИЕ #3: Термин «CRM» отпугивает целевую аудиторию
ИНСАЙТЫ: [2 инсайта из кластера Е]
БИЗНЕС-МОДЕЛЬ: КЗ
СЛЕДСТВИЯ:
- Ребрендинг: «Умная база клиентов»
- Контент-маркетинг без слова «CRM»
- SEO: long-tail запросы («как вести клиентов», не «CRM»)
```

## Типичные ошибки при картировании инсайтов

### Ошибка 1: Преждевременная фильтрация

**Проблема:** Команда отбрасывает «неудобные» инсайты, которые противоречат гипотезам.

```
НЕПРАВИЛЬНО: «Этот инсайт не вписывается в нашу концепцию — уберём»
ПРАВИЛЬНО: «Этот инсайт противоречит концепции — что это значит?»
```

**Почему это важно:** Противоречия — самые ценные находки. Они указывают на слепые зоны или необходимость пивота.

### Ошибка 2: Кластеризация по источнику, а не по смыслу

**Проблема:** Инсайты группируются по тому, от кого получены, а не по содержанию.

```
НЕПРАВИЛЬНО:
- Кластер «Анна»: [все инсайты от Анны]
- Кластер «Сергей»: [все инсайты от Сергея]

ПРАВИЛЬНО:
- Кластер «Страх потери данных»: [инсайты от Анны, Сергея, Михаила]
- Кластер «Нет времени»: [инсайты от Анны, Сергея]
```

### Ошибка 3: Отсутствие связи с бизнес-моделью

**Проблема:** Инсайты синтезируются, но не связываются с конкретными элементами бизнес-модели.

**Последствия:** На этапе проектирования команда не знает, что делать с этими знаниями.

**Решение:** Каждое знание должно содержать секцию «Связь с бизнес-моделью» с конкретными элементами.

### Ошибка 4: Слишком общие формулировки знаний

**Проблема:** Знания сформулированы абстрактно, их невозможно применить.

```
НЕПРАВИЛЬНО: «Клиентам важна простота»
ПРАВИЛЬНО: «Клиенты отказываются от продукта, если onboarding требует
более 3 шагов. Ожидание: начать работу за 30 секунд без регистрации»
```

**Тест actionability:** Если знание нельзя превратить в конкретное требование к продукту — оно слишком абстрактное.

### Ошибка 5: Игнорирование пробелов

**Проблема:** Команда фокусируется на том, что узнали, и не замечает, чего не хватает.

По данным исследования продуктовых команд, 65% провалов связаны не с неправильными инсайтами, а с пропущенными областями исследования.

**Решение:** После картирования проверить: все ли элементы бизнес-модели покрыты? Если нет — вернуться к наблюдению.

## Чеклист качества процесса изучения

**Результаты процесса И:**

- [ ] Все инсайты из наблюдений собраны (ни один не потерян)
- [ ] Инсайты сгруппированы по смыслу, а не по источнику
- [ ] Каждый кластер имеет ёмкое название
- [ ] Связи между кластерами визуализированы
- [ ] Инсайты отображены на элементы бизнес-модели (матрица)
- [ ] Паттерны идентифицированы (повторы, противоречия, связи)
- [ ] Пробелы выявлены (какие элементы не покрыты)
- [ ] Знания сформулированы в стандартном формате
- [ ] Каждое знание actionable (можно превратить в требование)

**Критерии готовности к переходу на этап проектирования:**

- [ ] Сформулировано минимум 5-7 ключевых знаний
- [ ] Знания покрывают минимум 70% элементов бизнес-модели
- [ ] Противоречия разрешены или превращены в гипотезы
- [ ] Пробелы закрыты (дополнительным наблюдением) или приняты как риски

## Часто задаваемые вопросы

### Сколько инсайтов нужно для качественного картирования?

Минимум 15-20 инсайтов из наблюдений для одного стратегического фокуса. При меньшем количестве паттерны не проявляются, кластеры получаются слишком мелкими. При количестве более 50 инсайтов рекомендуется разделить на подгруппы по элементам бизнес-модели.

### Можно ли автоматизировать кластеризацию инсайтов?

Частично. AI-инструменты (тематическое моделирование, кластеризация текстов) могут предложить первичную группировку. Однако финальное наименование кластеров и интерпретация связей требуют человеческого суждения — алгоритмы не понимают бизнес-контекст.

### Что делать, если инсайты противоречат друг другу?

Противоречия — не проблема, а ценная находка. Три варианта: (1) разные сегменты клиентов с разными потребностями — сегментировать; (2) разные контексты использования — разные сценарии; (3) клиенты говорят одно, делают другое — доверять поведению, не словам.

### Как понять, что пора переходить к проектированию?

Признаки готовности: (1) новые инсайты подтверждают существующие знания, но не добавляют новых; (2) матрица отображения покрывает 70%+ элементов бизнес-модели; (3) команда может ответить на вопрос «почему клиент будет платить» с опорой на конкретные знания.

## Выводы

- **Картирование — мост между исследованиями и действиями.** Без систематической трансформации инсайтов в знания исследования остаются отчётами, а не основой для решений.

- **Кластеризуйте по смыслу, не по источнику.** Группировка по персонам или интервью скрывает паттерны. Дайте кластерам проявиться из данных.

- **Связывайте каждый инсайт с бизнес-моделью.** Матрица отображения превращает абстрактные наблюдения в конкретные направления для проектирования.

- **Противоречия — самые ценные находки.** Не отбрасывайте «неудобные» инсайты. Они указывают на сегментацию, нестандартные решения или слепые зоны.
