## Что такое валидация в DTS

Валидация — это восьмой этап методологии **дизайн-мышления для стратегии (Design Thinking for Strategy, DTS)** и четвёртый на уровне бизнес-модели. На предыдущем этапе (Проектирование) команда создала 2-3 прототипа бизнес-модели. Теперь задача — проверить ключевые допущения и выбрать финальный вариант.

### Место в процессе дизайн-мышления для стратегии

```
ФУНДАМЕНТАЛЬНЫЙ УРОВЕНЬ
Б:Бриф → С:Среда → Т:Тренды → Ф:Фокус
                                  │
                          КТ1: Утверждение фокуса
                                  │
                                  ▼
УРОВЕНЬ БИЗНЕС-МОДЕЛИ
Н:Наблюдение → И:Изучение → П:Проектирование → В:ВАЛИДАЦИЯ
       ▲                                            │
       └──────────── итерации ──────────────────────┘
                                                    │
                                    КТ2: Утверждение бизнес-модели
```

Процесс валидации (В) — конвергентный этап. Если проектирование — дивергенция (генерация вариантов), то валидация — конвергенция (проверка и выбор). После валидации команда переходит к контрольной точке 2, где лица, принимающие решения (Decision Takers), утверждают финальную бизнес-модель.

### Валидация vs традиционные исследования рынка

Валидация стратегических гипотез в DTS кардинально отличается от классических маркетинговых исследований. Ключевые различия:

| Традиционные исследования | Валидация в DTS |
|---------------------------|-----------------|
| Изучение рынка «в целом» | Проверка конкретных допущений |
| Фокус-группы и опросы | Эксперименты с реальным поведением |
| Длительные проекты (месяцы) | Быстрые итерации (дни/недели) |
| Отчёт с рекомендациями | Решение: сохранить / изменить / отбросить |
| «Что думают клиенты» | «Как клиенты поступают» |

Согласно статье «[Проблемы валидации продуктовых гипотез](https://cyberleninka.ru/article/n/problemy-validatsii-produktovyh-gipotez-i-instrumenty-pomogayuschie-preodolet-ih)» (Окусков, Сахаров, 2021), ключевая проблема традиционных методов — разрыв между декларируемыми намерениями и реальным поведением. Люди говорят одно, а делают другое. Валидация в DTS фокусируется на поведении.

### Почему валидация критична для стратегии

По данным исследований стартапов, причина №1 провала — «отсутствие рыночного спроса» (35-42%). Эта проблема актуальна не только для стартапов, но и для корпоративных стратегий.

> «Ни один бизнес-план не выдерживает первого контакта с клиентом» — Стив Бланк, автор методологии Customer Development

Валидация позволяет обнаружить критические ошибки до масштабных инвестиций. По данным [Lean Startup](https://theleanstartup.com/), подход MVP снижает начальные затраты на разработку до 60%. Тот же принцип применим к стратегии: проверяем допущения дешёвыми экспериментами, а не дорогостоящей реализацией.

## Пять шагов процесса валидации

Процесс валидации в DTS состоит из пяти последовательных шагов.

```
В:1 ─── Идентификация допущений
        │
        └── Выявление всех допущений в прототипах

В:2 ─── Приоритизация допущений
        │
        └── Фокус на критичных и неопределённых

В:3 ─── Разработка экспериментов
        │
        └── Быстрые, дешёвые, фальсифицируемые тесты

В:4 ─── Проведение экспериментов
        │
        └── Тестирование с реальными клиентами

В:5 ─── Анализ результатов
        │
        └── Решение: сохранить / изменить / отбросить
```

### Шаг В:1 — Идентификация допущений

На первом шаге команда выявляет все допущения, заложенные в прототип бизнес-модели. Каждое решение в прототипе — это гипотеза, которая может оказаться ложной.

**Для каждого из 15 элементов бизнес-модели сформулируйте допущения:**

| Элемент | Пример допущения | Тип |
|---------|------------------|-----|
| КС (Сегменты) | «Сегмент X существует и достаточно велик (>10 000 компаний)» | Существование |
| КЗ (Задачи) | «Клиенты тратят >2 часов в день на задачу Y» | Потребность |
| ПЦП (Ценность) | «Клиенты готовы платить за решение проблемы Z» | Желательность |
| ККП (Активности) | «Мы можем создать технологию W за 6 месяцев» | Выполнимость |
| ФД (Доходы) | «Unit-экономика сходится при цене P и CAC < X» | Жизнеспособность |

**Пять типов допущений:**

1. **Допущения существования** — сегменты, ресурсы, каналы действительно существуют
2. **Допущения потребности** — клиенты действительно имеют проблему (jobs-to-be-done)
3. **Допущения желательности** — клиенты хотят наше решение и готовы платить
4. **Допущения выполнимости** — мы можем создать продукт/услугу
5. **Допущения жизнеспособности** — модель будет прибыльной

**Формат записи допущения:**

```
Элемент:    КС (Сегменты клиентов)
Допущение:  Микробизнес (1-5 сотрудников) в России насчитывает >500 000 компаний
Источник:   Знание И:3 из этапа изучения
Тип:        Существование
Риск:       Средний (данные Росстата могут быть неполными)
```

### Шаг В:2 — Приоритизация допущений

После идентификации команда получает десятки допущений. Проверять все — неэффективно. Нужна приоритизация по двум критериям: важность для стратегии и уровень неопределённости.

**Матрица приоритизации допущений:**

```
                    ВАЖНОСТЬ ДЛЯ СТРАТЕГИИ
                    Низкая         Высокая
                ┌──────────────┬──────────────┐
        Высокая │   Отложить   │  ПРИОРИТЕТ 1 │
НЕОПРЕ-         │              │  (Leap of    │
ДЕЛЁН-          ├──────────────┤   Faith)     │
НОСТЬ   Низкая  │   Игнорить   ├──────────────┤
                │              │  ПРИОРИТЕТ 2 │
                └──────────────┴──────────────┘
```

**Правило «Прыжок веры» (Leap of Faith):** Начинать с допущений, которые могут опровергнуть всю стратегию. Это допущения высокой важности и высокой неопределённости. Если они ложны — вся бизнес-модель рушится.

**Примеры приоритизации:**

| Допущение | Важность | Неопределённость | Приоритет |
|-----------|----------|------------------|-----------|
| «Клиенты готовы платить 990 руб/мес» | Высокая | Высокая | 1 |
| «WhatsApp API позволяет создать бота» | Высокая | Низкая | 2 |
| «Офис можно арендовать в центре» | Низкая | Низкая | — |

### Шаг В:3 — Разработка экспериментов

Для каждого приоритетного допущения команда разрабатывает эксперимент. Хороший эксперимент должен соответствовать четырём критериям.

**Критерии хорошего эксперимента:**

1. **Быстрый** — результат за часы или дни, не за недели
2. **Дешёвый** — минимальные затраты ресурсов
3. **Понятный** — чёткий критерий успеха/неудачи
4. **Фальсифицируемый** — можно получить результат «гипотеза опровергнута»

**Типы экспериментов для разных категорий допущений:**

| Тип допущения | Методы валидации | Пример |
|---------------|------------------|--------|
| **Существование** | Анализ данных, кабинетное исследование | Проверка объёма сегмента по Росстату |
| **Потребность** | Проблемные интервью, JTBD-интервью | 20 интервью с целевыми клиентами |
| **Желательность** | Лендинг с формой, fake door test | Страница с предзаказом |
| **Выполнимость** | Прототип, MVP, техническая проверка концепции | Работающий прототип ключевой функции |
| **Жизнеспособность** | Финансовая модель, тест цены | A/B тест с разными ценами |

По данным статьи о [Customer Development](https://vc.ru/marketing/575262-customer-development-ili-kastdev-custdev-v-biznese-instrukciya-s-primerami), для валидации потребностей достаточно 7-30 проблемных интервью, чтобы выявить устойчивые паттерны.

### Шаг В:4 — Проведение экспериментов

На четвёртом шаге команда проводит эксперименты с реальными клиентами или данными. Ключевое правило — фиксировать всё по формату.

**Формат карточки эксперимента:**

```
ГИПОТЕЗА:       Микробизнес (салоны красоты) готов платить 990 руб/мес
                за CRM в WhatsApp

МЕТОД:          Лендинг с формой предзаказа + таргетированная реклама

КРИТЕРИЙ:       ≥5% конверсия из посетителя в заявку

РАЗМЕР:         1000 посетителей лендинга

БЮДЖЕТ:         15 000 руб на рекламу

СРОК:           7 дней

─────────────────────────────────────────────────────────────

РЕЗУЛЬТАТ:      Конверсия 7,3% (73 заявки из 1000 посетителей)

ВЫВОД:          ПОДТВЕРЖДЕНО

СЛЕДУЮЩИЙ ШАГ:  Перейти к валидации готовности платить (тест цены)
```

**Правила проведения экспериментов:**

1. **Не менять критерий после эксперимента.** Соблазн сдвинуть планку после получения результата искажает валидацию.

2. **Документировать всё.** Даже «неудачные» эксперименты — ценная информация для следующих итераций.

3. **Тестировать поведение, не мнения.** «Я бы купил» ≠ «Я купил». Лендинг с формой предзаказа надёжнее опроса.

4. **Один эксперимент — одна гипотеза.** Комбинированные тесты не дают понять, что именно сработало.

### Шаг В:5 — Анализ результатов

Финальный шаг — анализ результатов и принятие решения о каждом допущении и прототипе в целом.

**Дерево решений по результатам эксперимента:**

```
                    Результат эксперимента
                            │
              ┌─────────────┼─────────────┐
              │             │             │
        ПОДТВЕРЖДЕНО   НЕОПРЕДЕЛЁННО  ОПРОВЕРГНУТО
              │             │             │
              ▼             ▼             ▼
        Сохранить      Провести      ┌────┴────┐
        элемент        ещё тест      │         │
                                  Изменить  Отбросить
                                  элемент   прототип
```

**Три возможных исхода:**

1. **Подтверждено** — результат соответствует или превышает критерий. Допущение сохраняется, переходим к следующему.

2. **Неопределённо** — результат близок к критерию, но не достигает его. Требуется дополнительный эксперимент с большей выборкой или другим методом.

3. **Опровергнуто** — результат значительно ниже критерия. Два варианта: изменить элемент бизнес-модели или отбросить весь прототип.

**Когда отбрасывать прототип:**

- Опровергнуто допущение типа «Прыжок веры» (высокая важность + высокая неопределённость)
- Опровергнуто >3 связанных допущений
- Изменение требует переделки >50% элементов модели

## Три измерения валидации

Каждый прототип бизнес-модели должен быть валидирован по трём измерениям. Провал в любом из них означает провал стратегии.

```
                    ┌─────────────────┐
                    │  ЖЕЛАТЕЛЬНОСТЬ  │
                    │  (Desirability) │
                    │                 │
                    │ Хотят ли клиенты│
                    │ это решение?    │
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
              ▼              │              ▼
    ┌─────────────────┐      │    ┌─────────────────┐
    │  ВЫПОЛНИМОСТЬ   │      │    │ ЖИЗНЕСПОСОБНОСТЬ│
    │  (Feasibility)  │◄─────┴───►│  (Viability)    │
    │                 │           │                 │
    │ Можем ли мы это │           │ Будет ли это    │
    │ сделать?        │           │ прибыльным?     │
    └─────────────────┘           └─────────────────┘
```

**Порядок валидации измерений:**

1. **Сначала желательность.** Если клиенты не хотят решение, остальное не имеет значения. По данным CB Insights, 42% стартапов провалились именно из-за отсутствия желательности.

2. **Затем выполнимость.** Можем ли мы создать то, что хотят клиенты? Есть ли технологии, компетенции, ресурсы?

3. **В конце жизнеспособность.** Сходится ли экономика? Готовы ли клиенты платить достаточно?

**Методы валидации по измерениям:**

| Измерение | Вопрос | Методы |
|-----------|--------|--------|
| **Желательность** | Хотят ли клиенты это? | Проблемные интервью, JTBD, лендинги, fake door |
| **Выполнимость** | Можем ли мы это сделать? | Прототипы, MVP, технические спайки, пилоты |
| **Жизнеспособность** | Будет ли это прибыльным? | Финмодели, тесты цен, unit-экономика |

## Практический пример: валидация для B2B SaaS

Продолжим пример из предыдущих статей. CRM-система для микробизнеса (прототип А: «WhatsApp-first CRM»).

**Исходный прототип:**

| Элемент | Содержание |
|---------|------------|
| КС | Микробизнес (1-5 сотрудников): салоны, автосервисы |
| КЗ | Не терять контакты клиентов; напоминать о визитах |
| ПЦП | «База клиентов в привычном мессенджере» |
| ППУ | Бот WhatsApp Business API |
| ФД | Подписка 990 руб/мес на сотрудника |

**В:1 — Идентификация допущений:**

| # | Допущение | Тип | Риск |
|---|-----------|-----|------|
| 1 | Салоны красоты теряют >10% клиентов из-за забытых контактов | Потребность | Высокий |
| 2 | Владельцы салонов готовы использовать бота вместо Excel | Желательность | Высокий |
| 3 | WhatsApp Business API позволяет создать нужную функциональность | Выполнимость | Средний |
| 4 | Владельцы готовы платить 990 руб/мес за каждого сотрудника | Жизнеспособность | Высокий |
| 5 | В России >100 000 салонов красоты с 1-5 сотрудниками | Существование | Низкий |

**В:2 — Приоритизация:**

```
                    ВАЖНОСТЬ
                    Низкая         Высокая
                ┌──────────────┬──────────────┐
        Высокая │              │  #1, #2, #4  │ ← Приоритет 1
НЕОПРЕ-         │              │              │
ДЕЛЁН-          ├──────────────┼──────────────┤
НОСТЬ   Низкая  │      #5      │      #3      │ ← Приоритет 2
                └──────────────┴──────────────┘
```

**В:3 — Разработка экспериментов:**

| Допущение | Эксперимент | Критерий | Бюджет | Срок |
|-----------|-------------|----------|--------|------|
| #1 Потеря клиентов | 15 проблемных интервью с владельцами салонов | ≥10 из 15 подтверждают проблему | 0 руб | 2 недели |
| #2 Готовность к боту | Лендинг + реклама в Instagram | ≥3% конверсия в заявку | 20 000 руб | 1 неделя |
| #4 Готовность платить | Звонки по заявкам с предложением предоплаты | ≥20% готовы внести депозит | 0 руб | 1 неделя |

**В:4 — Результаты экспериментов:**

| # | Результат | Вывод |
|---|-----------|-------|
| #1 | 12 из 15 подтвердили проблему потери клиентов | ПОДТВЕРЖДЕНО |
| #2 | Конверсия 4,7% (94 заявки из 2000) | ПОДТВЕРЖДЕНО |
| #4 | 8 из 94 (8,5%) внесли депозит 500 руб | НЕОПРЕДЕЛЁННО |

**В:5 — Анализ и решение:**

Допущения #1 и #2 подтверждены. Допущение #4 требует уточнения: готовность платить ниже ожиданий.

**Решение:** Провести дополнительный эксперимент — A/B тест с ценами 490, 790, 990 руб/мес.

**Дополнительный эксперимент:**

| Цена | Конверсия в депозит |
|------|---------------------|
| 490 руб | 24% |
| 790 руб | 15% |
| 990 руб | 8% |

**Финальное решение:** Изменить элемент ФД на 790 руб/мес. При 15% конверсии и CAC 2000 руб unit-экономика сходится.

## Типичные ошибки при валидации

### Ошибка 1: Валидация мнений вместо поведения

**Проблема:** Команда спрашивает «Вы бы купили это?» вместо создания условий для реальной покупки.

```
НЕПРАВИЛЬНО: «80% респондентов сказали, что купили бы»
ПРАВИЛЬНО: «15% посетителей лендинга внесли депозит»
```

По данным исследований, декларируемые намерения и реальное поведение расходятся в 3-5 раз. Люди переоценивают свою готовность платить.

**Решение:** Тестируйте поведение. Лендинг с формой предзаказа, fake door test, минимальный депозит — всё, что требует действия от клиента.

### Ошибка 2: Слишком маленькая выборка

**Проблема:** Выводы на основе 3-5 интервью или 50 посетителей лендинга.

**Последствия:** Результат может быть случайным. Нельзя различить сигнал и шум.

**Минимальные размеры выборки:**

| Метод | Минимум | Рекомендуется |
|-------|---------|---------------|
| Проблемные интервью | 10 | 15-20 |
| Лендинг (конверсия) | 200 | 500-1000 |
| A/B тест | 100 на вариант | 300+ на вариант |
| Тест цены | 50 | 100+ |

### Ошибка 3: Сдвиг критерия после эксперимента

**Проблема:** Критерий успеха был «≥10%», получили 7%, решили что «7% тоже неплохо».

```
НЕПРАВИЛЬНО:
Критерий: ≥10%
Результат: 7%
Вывод: «Почти достигли, считаем подтверждённым»

ПРАВИЛЬНО:
Критерий: ≥10%
Результат: 7%
Вывод: «Не достигли → неопределённо → нужен ещё эксперимент»
```

**Решение:** Фиксируйте критерий ДО эксперимента. Не меняйте его после получения результата. Если результат неопределённый — проведите дополнительный тест.

### Ошибка 4: Игнорирование негативных результатов

**Проблема:** Команда эмоционально привязана к прототипу и ищет оправдания опровергнутым допущениям.

**Признаки:**
- «Это просто неправильная выборка»
- «Мы плохо объяснили ценность»
- «Рынок ещё не готов»

**Решение:** Опровергнутое допущение — это ценная информация, которая экономит деньги. Лучше узнать об ошибке на этапе эксперимента за 20 000 руб, чем после запуска за 2 000 000 руб.

### Ошибка 5: Валидация в неправильном порядке

**Проблема:** Команда начинает с выполнимости (строит MVP), не проверив желательность (хотят ли клиенты).

```
НЕПРАВИЛЬНО: Выполнимость → Желательность → Жизнеспособность
ПРАВИЛЬНО: Желательность → Выполнимость → Жизнеспособность
```

**Последствия:** Построен продукт, который никому не нужен. 42% стартапов провалились именно так.

**Решение:** Всегда начинайте с желательности. Проблемные интервью и лендинги стоят дешевле, чем разработка MVP.

## Чеклист качества процесса валидации

**Результаты процесса В:**

- [ ] Все допущения прототипа идентифицированы (≥10 допущений)
- [ ] Допущения классифицированы по 5 типам
- [ ] Допущения приоритизированы по матрице (важность × неопределённость)
- [ ] Выявлены 3-5 допущений типа «Прыжок веры»
- [ ] Для каждого приоритетного допущения разработан эксперимент
- [ ] Эксперименты соответствуют 4 критериям (быстрый, дешёвый, понятный, фальсифицируемый)
- [ ] Критерии успеха зафиксированы ДО проведения экспериментов
- [ ] Эксперименты проведены с достаточной выборкой
- [ ] Результаты задокументированы в формате карточек
- [ ] По каждому допущению принято решение (сохранить / изменить / отбросить)

**Критерии готовности к контрольной точке 2:**

- [ ] Все допущения типа «Прыжок веры» подтверждены
- [ ] Валидированы три измерения: желательность, выполнимость, жизнеспособность
- [ ] Прототип бизнес-модели финализирован с учётом результатов экспериментов
- [ ] Команда согласна с финальной версией
- [ ] Подготовлена презентация для лиц, принимающих решения

## Часто задаваемые вопросы

### Сколько экспериментов нужно провести?

Количество экспериментов зависит от числа приоритетных допущений. Обычно это 5-10 экспериментов для одного прототипа. Фокусируйтесь на допущениях «Прыжок веры» — высокая важность и высокая неопределённость. Без их подтверждения нет смысла проверять остальные.

### Как быстро можно провести валидацию?

При правильной организации — 2-4 недели на один прототип. Проблемные интервью: 1-2 недели (15-20 интервью). Лендинг с рекламой: 1 неделя. Тест цены: 1 неделя. Параллельное проведение экспериментов сокращает общее время.

### Что делать, если все прототипы провалились?

Вернуться на этап проектирования (П) или изучения (И). Провал всех прототипов означает, что знания о клиентах недостаточны или стратегический фокус выбран неверно. Это нормальная часть итеративного процесса DTS. Лучше узнать это сейчас, чем после запуска.

### Можно ли пропустить валидацию для «очевидных» допущений?

Только если допущение имеет низкую неопределённость И подтверждено надёжными данными (официальная статистика, собственный опыт компании). «Очевидность» субъективна. 42% стартапов были уверены в наличии рынка — и ошиблись. При сомнениях — проверяйте.

## Выводы

- **Проверяйте допущения, не мнения.** Валидация фокусируется на реальном поведении клиентов, а не на декларируемых намерениях. Лендинг с депозитом надёжнее опроса «купили бы вы».

- **Начинайте с допущений «Прыжок веры».** Приоритизируйте по матрице важность × неопределённость. Если ключевое допущение ложно — вся стратегия рушится.

- **Желательность → Выполнимость → Жизнеспособность.** Валидируйте в правильном порядке. Сначала проверьте, что клиенты хотят решение, потом — что вы можете его создать.

- **Фиксируйте критерии заранее.** Не сдвигайте планку после получения результата. Неопределённый результат — повод для нового эксперимента, а не для самообмана.
