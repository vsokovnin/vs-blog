## Что такое проектирование в DTS

Проектирование — это седьмой этап методологии **дизайн-мышления для стратегии (Design Thinking for Strategy, DTS)** и третий на уровне бизнес-модели. На предыдущем этапе (Изучение) команда сформулировала знания о клиентах, рынке и компетенциях. Теперь задача — превратить эти знания в конкретные варианты бизнес-модели.

### Место в процессе дизайн-мышления для стратегии

```
ФУНДАМЕНТАЛЬНЫЙ УРОВЕНЬ
Б:Бриф → С:Среда → Т:Тренды → Ф:Фокус
                                  │
                          КТ1: Утверждение фокуса
                                  │
                                  ▼
УРОВЕНЬ БИЗНЕС-МОДЕЛИ
Н:Наблюдение → И:Изучение → П:ПРОЕКТИРОВАНИЕ → В:Валидация
       ▲                            │
       └──────── итерации ──────────┘
                                    │
                            КТ2: Утверждение бизнес-модели
```

Процесс проектирования (П) — дивергентный этап. Если изучение — конвергенция (фильтрация знаний), то проектирование — дивергенция (генерация множества вариантов). Результаты проектирования передаются на этап валидации для проверки гипотез.

### Проектирование vs традиционное планирование

Прототипирование стратегии в DTS кардинально отличается от классического стратегического планирования. Ключевые различия:

| Традиционное планирование | Проектирование в DTS |
|---------------------------|---------------------|
| Один «правильный» план | 2-3 альтернативных прототипа |
| Детализация на годы вперёд | Минимально жизнеспособная модель |
| Фокус на прогнозах | Фокус на допущениях и их проверке |
| Длительная разработка (месяцы) | Быстрые итерации (дни/недели) |
| Провал = катастрофа | Провал = источник знаний |

Согласно статье «[Методологии проектирования стратегии бизнеса](https://cyberleninka.ru/article/n/metodologii-proektirovaniya-strategii-biznesa-ot-dizayna-produkta-k-proektirovaniyu-platform)», переход от единственного плана к множеству прототипов снижает риск стратегических ошибок, потому что команда не привязывается к одному решению эмоционально.

### Почему прототипирование критично для стратегии

По данным [Lean Startup](https://theleanstartup.com/), подход MVP (Minimum Viable Product) снижает начальные затраты на разработку до 60%. Тот же принцип работает для стратегии: вместо детальной проработки одного варианта команда создаёт несколько «лёгких» прототипов и проверяет их критические допущения.

> «Ни один бизнес-план не выдерживает первого контакта с клиентом» — Стив Бланк, автор методологии Customer Development

Прототип бизнес-модели — это гипотеза, сформулированная достаточно детально для проверки, но не настолько, чтобы её было жалко выбросить.

## Детализированная бизнес-модель: 15 элементов

Прежде чем перейти к процессу проектирования, разберём структуру детализированной бизнес-модели. Она расширяет классический Business Model Canvas Александра Остервальдера до 15 элементов.

### Четыре компонента бизнес-модели

```
┌─────────────────────────────────────────────────────────────────┐
│  ДИФФЕРЕНЦИАЦИЯ (верхняя граница)                               │
├─────────────────┬─────────────────┬─────────────────────────────┤
│                 │                 │                             │
│    КЛИЕНТЫ      │   ПРЕДЛОЖЕНИЯ   │       КОМПЕТЕНЦИИ           │
│                 │                 │                             │
├─────────────────┴─────────────────┴─────────────────────────────┤
│                          ФИНАНСЫ                                │
├─────────────────────────────────────────────────────────────────┤
│  ЛИДЕРСТВО ПО ИЗДЕРЖКАМ (нижняя граница)                        │
└─────────────────────────────────────────────────────────────────┘
```

**Ключевой принцип:** Успешная компания превосходит конкурентов в ОДНОМ компоненте (стратегический фокус, выбранный на этапе Ф) и остаётся конкурентоспособной в трёх других.

### 15 элементов детализированной модели

| Компонент | Код | Элемент | Ключевой вопрос |
|-----------|-----|---------|-----------------|
| **КЛИЕНТЫ** | КС | Сегменты клиентов | Для кого создаём ценность? |
| | КЗ | Задачи клиентов (JTBD) | Какие задачи решаем? |
| | КВ | Взаимоотношения | Какой тип отношений ожидают? |
| | КД | Каналы доставки | Как доставляем ценность? |
| **ПРЕДЛОЖЕНИЯ** | ПЦП | Ценностное предложение | Какую проблему решаем? |
| | ППУ | Продукты и услуги | Что конкретно предлагаем? |
| **КОМПЕТЕНЦИИ** | ККП | Активности для конкурентного преимущества | Что делаем лучше других? |
| | КЦП | Активности для ценового преимущества | Что делаем дешевле? |
| | КАА | Активности на аутсорсе | Что отдаём партнёрам? |
| | КРР | Расходуемые ресурсы | Что потребляем? |
| | ККР | Капитальные ресурсы | Чем владеем? |
| | КТР | Трудовые ресурсы | Кто работает? |
| | КНК | Новые навыки и компетенции | Чему нужно научиться? |
| **ФИНАНСЫ** | ФЗ | Структура затрат | На что тратим? |
| | ФД | Потоки доходов | За что платят? |

По данным [Value Proposition Design](https://www.strategyzer.com/library/the-value-proposition-canvas), детальная проработка связи между задачами клиентов (КЗ) и ценностным предложением (ПЦП) увеличивает вероятность product-market fit в 2,5 раза.

## Четыре шага процесса проектирования

Процесс проектирования в DTS состоит из четырёх последовательных шагов.

```
П:1 ─── Идеация
        │
        └── Генерация идей для каждого элемента
            бизнес-модели

П:2 ─── Прототипирование
        │
        └── Создание 2-3 вариантов бизнес-модели

П:3 ─── Проверка консистентности
        │
        └── Валидация связей между элементами

П:4 ─── Выбор прототипов для валидации
        │
        └── Приоритизация наиболее перспективных
```

### Шаг П:1 — Идеация

На первом шаге команда генерирует максимальное количество идей для каждого элемента бизнес-модели. Используйте знания, сформулированные на этапе изучения, как отправную точку.

**Правила брейнсторминга для стратегии:**

1. **Отложить суждение.** Не критиковать идеи во время генерации
2. **Поощрять дикие идеи.** Радикальные идеи часто ведут к прорывам
3. **Строить на идеях других.** «Да, и...» вместо «Нет, но...»
4. **Фокусироваться на количестве.** Цель: 50+ идей за сессию
5. **Оставаться на теме.** Возвращаться к стратегическому фокусу
6. **Быть визуальным.** Рисовать, а не только писать

**Техника антиконвенционального мышления:**

Для генерации прорывных идей используйте технику «переворачивания конвенций»:

```
ШАГ 1: Определить конвенции отрасли
       │
       ▼
ШАГ 2: Выбрать одну конвенцию для вызова
       │
       ▼
ШАГ 3: Перевернуть конвенцию
       │
       ▼
ШАГ 4: Проверить жизнеспособность перевернутой идеи
```

**Пример антиконвенционального мышления для B2B SaaS:**

| Конвенция отрасли | Перевёрнутая версия | Проверка |
|-------------------|---------------------|----------|
| CRM требует обучения | CRM не требует обучения (работает как WhatsApp) | Технически возможно через чат-ботов |
| Данные хранятся в облаке | Данные хранятся у клиента | Потеря функций синхронизации — не жизнеспособно |
| Подписка помесячно | Оплата за результат (за закрытую сделку) | Риск злоупотреблений — требует проработки |

### Шаг П:2 — Прототипирование бизнес-модели

После идеации команда собирает идеи в целостные прототипы бизнес-модели. Создайте 2-3 альтернативных варианта.

**Порядок заполнения зависит от стратегического фокуса:**

**Если фокус — КЛИЕНТЫ:**
```
КС → КЗ → КВ → КД → ПЦП → ППУ → КА → КР → ФЗ → ФД
```

**Если фокус — ПРЕДЛОЖЕНИЯ:**
```
ПЦП → ППУ → КС → КЗ → КВ → КД → КА → КР → ФЗ → ФД
```

**Если фокус — КОМПЕТЕНЦИИ:**
```
ККП → КРР → ПЦП → ППУ → КС → КЗ → КВ → КД → ФЗ → ФД
```

**Если фокус — ФИНАНСЫ:**
```
ФД → ФЗ → ПЦП → ППУ → КС → КЗ → КВ → КД → КА → КР
```

**Принцип:** Начинайте с элементов вашего стратегического фокуса. Они определяют, вокруг чего строится вся модель.

**Формат прототипа:**

Для каждого элемента заполните:
- **Гипотеза:** Что мы предполагаем?
- **Источник:** Какие знания из этапа И это подтверждают?
- **Риск:** Что может пойти не так?

### Шаг П:3 — Проверка консистентности

Третий шаг — убедиться, что все элементы бизнес-модели логически связаны и не противоречат друг другу.

**Карта связей между элементами:**

```
КС ◄──► КЗ ◄──► ПЦП ◄──► ППУ ◄──► КА ◄──► КР
         │                         │
         └────────► КВ ◄───────────┘
                    │
                    ▼
                   КД
                    │
                    ▼
              ФЗ ◄──► ФД
```

**Правила консистентности:**

1. **Каждое ценностное предложение (ПЦП) должно решать задачи клиентов (КЗ).** Если ПЦП не связано ни с одной задачей — уберите его.

2. **Каждый продукт/услуга (ППУ) должен создавать ценностное предложение.** Продукт без ценности — лишний элемент.

3. **Каждая ключевая активность (КА) должна поддерживать ППУ.** Активность, не ведущая к продукту — потеря ресурсов.

4. **Затраты (ФЗ) должны покрывать все КА и КР.** Забытые затраты — источник финансовых проблем.

5. **Доходы (ФД) должны превышать затраты.** Иначе модель нежизнеспособна.

**Чеклист консистентности:**

- [ ] Все КЗ связаны хотя бы с одним ПЦП
- [ ] Все ПЦП реализуются через ППУ
- [ ] Все ККП создают дифференциацию
- [ ] Все КЦП снижают затраты
- [ ] ФД > ФЗ при целевых объёмах

### Шаг П:4 — Выбор прототипов для валидации

Финальный шаг проектирования — выбрать 1-2 прототипа для тестирования на этапе валидации.

**Критерии приоритизации:**

| Критерий | Вопрос | Как оценить |
|----------|--------|-------------|
| **Желательность** | Клиенты хотят это? | Знания из этапа И |
| **Выполнимость** | Мы можем это сделать? | Анализ компетенций |
| **Жизнеспособность** | Это будет прибыльным? | Финансовая модель |
| **Отличительность** | Это отличается от конкурентов? | Анализ среды (этап С) |

**Матрица приоритизации прототипов:**

```
                    ПОТЕНЦИАЛ РЫНКА
                    Низкий         Высокий
                ┌──────────────┬──────────────┐
        Высокая │  Изучить     │  ПРИОРИТЕТ 1 │
ВЫПОЛ-          │  глубже      │              │
НИМОСТЬ         ├──────────────┼──────────────┤
        Низкая  │  Отложить    │  ПРИОРИТЕТ 2 │
                │              │  (требует    │
                │              │  инвестиций) │
                └──────────────┴──────────────┘
```

**Результат шага:** 1-2 прототипа бизнес-модели с чётко сформулированными допущениями для проверки.

## Практический пример: проектирование для B2B SaaS

Продолжим пример из предыдущих статей. CRM-система для малого бизнеса со стратегическим фокусом на клиентах.

**Исходные знания из этапа И:**

1. Безопасность данных критичнее функциональности
2. WhatsApp — единственный допустимый интерфейс
3. Термин «CRM» отпугивает целевую аудиторию
4. Нет времени на обучение — продукт должен быть интуитивным
5. Нужен контроль сотрудников, но простой

**П:1 — Идеация (примеры идей по элементам):**

| Элемент | Идеи |
|---------|------|
| КС (Сегменты) | Салоны красоты; Автосервисы; Дистрибьюторы продуктов |
| КЗ (Задачи) | Не терять клиентов; Напоминать о записи; Контролировать менеджеров |
| ПЦП (Ценность) | «Клиенты никогда не забываются»; «База всегда под рукой в телефоне» |
| ППУ (Продукт) | Бот в WhatsApp; Голосовой ввод; OCR визиток |
| КВ (Отношения) | Самообслуживание; Чат-поддержка; Личный менеджер |

**П:2 — Два прототипа бизнес-модели:**

**Прототип А: «WhatsApp-first CRM»**

| Компонент | Содержание |
|-----------|------------|
| КС | Микробизнес (1-5 сотрудников): салоны, автосервисы |
| КЗ | Не терять контакты клиентов; Напоминать о визитах |
| ПЦП | «База клиентов в привычном мессенджере» |
| ППУ | Бот WhatsApp Business API |
| КВ | Полное самообслуживание |
| КД | Telegram-каналы, рекомендации, Яндекс.Директ |
| ККП | Интеграция с WhatsApp Business API |
| КЦП | Zero onboarding (без обучения) |
| ФД | Подписка 990 руб/мес на сотрудника |
| ФЗ | API WhatsApp, серверы, поддержка |

**Прототип Б: «Умная база клиентов»**

| Компонент | Содержание |
|-----------|------------|
| КС | Малый бизнес (5-20 сотрудников): дистрибьюторы, агентства |
| КЗ | Контролировать работу менеджеров; Не терять сделки |
| ПЦП | «Контроль продаж без сложностей CRM» |
| ППУ | Веб-приложение + мобильное приложение |
| КВ | Чат-поддержка + обучающие видео |
| КД | Контент-маркетинг (без слова «CRM»), SEO |
| ККП | UX уровня B2C-приложений |
| КЦП | Автоматизация отчётов |
| ФД | Подписка 2490 руб/мес + % от интеграций |
| ФЗ | Разработка, контент, поддержка |

**П:3 — Проверка консистентности:**

**Прототип А:**
- [x] КЗ → ПЦП: «Не терять контакты» → «База в мессенджере» ✓
- [x] ПЦП → ППУ: Бот реализует ценностное предложение ✓
- [x] ФД > ФЗ: 990 руб × 10 000 пользователей = 9,9 млн, затраты API ~2 млн ✓

**Прототип Б:**
- [x] КЗ → ПЦП: «Контролировать менеджеров» → «Контроль без сложностей» ✓
- [ ] ФД > ФЗ: Требует расчёта затрат на разработку двух платформ ⚠️

**П:4 — Выбор:**

| Критерий | Прототип А | Прототип Б |
|----------|------------|------------|
| Желательность | Высокая (подтверждено знаниями) | Средняя |
| Выполнимость | Высокая (меньше разработки) | Средняя (два клиента) |
| Жизнеспособность | Высокая | Требует расчёта |
| Отличительность | Высокая (уникальный канал) | Низкая (много конкурентов) |

**Решение:** Прототип А идёт на валидацию первым. Прототип Б — в резерв.

## Типичные ошибки при проектировании бизнес-модели

### Ошибка 1: Один «идеальный» план

**Проблема:** Команда создаёт один детальный план вместо нескольких прототипов.

```
НЕПРАВИЛЬНО: 1 план × 100% детализации
ПРАВИЛЬНО: 3 прототипа × 30% детализации каждый
```

**Последствия:** Эмоциональная привязанность к единственному варианту. При провале — крах всей стратегии.

**Решение:** Всегда создавайте минимум 2-3 альтернативных прототипа.

### Ошибка 2: Игнорирование стратегического фокуса

**Проблема:** Заполнение бизнес-модели без учёта выбранного фокуса — начинают с финансов или с продукта.

По данным исследований стратегических сессий, 60% команд начинают проектирование с «удобных» элементов (продукт, доходы), а не с элементов фокуса.

**Решение:** Строго следуйте порядку заполнения в зависимости от фокуса. Если фокус — клиенты, начинайте с КС, не с ПЦП.

### Ошибка 3: Несвязанные элементы

**Проблема:** Элементы бизнес-модели заполнены, но не проверены на консистентность.

```
НЕПРАВИЛЬНО:
ПЦП: «Лучший сервис на рынке»
КА: «Минимизация затрат на поддержку»
→ Противоречие!

ПРАВИЛЬНО:
ПЦП: «Лучший сервис на рынке»
КА: «Инвестиции в обучение поддержки»
→ Консистентно
```

**Решение:** После заполнения пройдите по карте связей и убедитесь, что каждый элемент логически связан с соседними.

### Ошибка 4: Отсутствие допущений

**Проблема:** Прототип описывает «что», но не «почему мы верим, что это сработает».

**Последствия:** На этапе валидации непонятно, какие гипотезы проверять.

**Решение:** Для каждого критического элемента формулируйте допущение:

```
Элемент: КС = микробизнес (1-5 сотрудников)
Допущение: Сегмент достаточно велик (>100,000 компаний в России)
Источник знания: И:3 (анализ рынка)
Риск: Сегмент может быть меньше
```

### Ошибка 5: Преждевременная детализация

**Проблема:** Команда детализирует прототип до уровня проектной документации.

**Последствия:** Потеря времени на элементы, которые будут изменены после валидации.

**Решение:** Прототип — это гипотеза, а не план реализации. Достаточно уровня детализации, позволяющего сформулировать допущения для проверки.

## Чеклист качества процесса проектирования

**Результаты процесса П:**

- [ ] Идеация проведена для каждого из 15 элементов
- [ ] Сгенерировано минимум 30 идей (суммарно по всем элементам)
- [ ] Созданы 2-3 прототипа бизнес-модели
- [ ] Все 15 элементов заполнены для каждого прототипа
- [ ] Элементы заполнены в порядке стратегического фокуса
- [ ] Связи между элементами проверены на консистентность
- [ ] Для критических элементов сформулированы допущения
- [ ] Прототипы приоритизированы по 4 критериям
- [ ] Выбран 1-2 прототипа для валидации

**Критерии готовности к переходу на этап валидации:**

- [ ] Выбранный прототип имеет минимум 5-7 явных допущений
- [ ] Допущения ранжированы по важности и неопределённости
- [ ] Команда согласна с приоритизацией
- [ ] Определены критерии «провала» для каждого допущения

## Часто задаваемые вопросы

### Сколько прототипов нужно создать?

Оптимально 2-3 прототипа. Меньше двух — нет альтернатив для сравнения. Больше трёх — размывается фокус и растут затраты на валидацию. Прототипы должны быть существенно разными, а не вариациями одной идеи.

### Как быстро можно создать прототип бизнес-модели?

При наличии качественных знаний из этапа изучения — за 2-4 часа командной работы. Один прототип на одну рабочую сессию. Не стремитесь к совершенству — это гипотеза для проверки, не финальный план.

### Что делать, если элементы противоречат друг другу?

Противоречие — сигнал о проблеме в логике модели. Три варианта: (1) изменить один из элементов; (2) разделить на два разных прототипа; (3) вернуться к этапу изучения за дополнительными знаниями.

### Когда переходить к валидации?

Признаки готовности: (1) есть 1-2 приоритизированных прототипа; (2) для каждого сформулированы ключевые допущения; (3) допущения можно проверить за разумное время (недели, не месяцы); (4) команда согласна с приоритизацией.

## Выводы

- **Прототипы, не планы.** Создавайте 2-3 альтернативных варианта бизнес-модели вместо одного «идеального» плана. Это снижает риск и эмоциональную привязанность.

- **Начинайте с фокуса.** Заполняйте бизнес-модель в порядке, определяемом стратегическим фокусом. Фокус на клиентах — начинайте с клиентов.

- **Проверяйте связи.** Консистентность элементов критична. Несвязанная модель развалится при реализации.

- **Формулируйте допущения.** Прототип без явных допущений невозможно валидировать. Каждое критическое решение — это гипотеза для проверки.
